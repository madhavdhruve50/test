Prompt: ETL Mapping Spec Automation Application Build

Act as a Senior Data Architect, Senior Data Analyst, Reverse Mortgage SME, and Python Solution Architect.

Build a reusable Python backend and Streamlit frontend application named:

ETL Mapping Spec Automation

Objective

The application should generate a business-ready ETL Mapping Specification Document from a metadata workbook generated by reverse-engineering an Excel report.

The final output should help data engineers, data analysts, ETL developers, and business SMEs understand how each report column is sourced, transformed, and populated.

Important Input Design

During the initial development phase only, I will provide:

1. Sample metadata workbooks
2. Sample target/output ETL mapping specification sheets

These sample inputs should be used to understand the metadata structure, expected output format, target object selection logic, transformation wording style, source object identification, source column mapping, row-level granularity, and validation expectations.

However, once the application is built, the final Streamlit application should take only one runtime input:

1. Metadata Workbook Upload

The sample target/output sheets should not be required every time the application runs. They should only be used during development to build reusable logic and output rules.

Power_Query_M_Code

Use this as the primary source for lineage and transformation logic.

The application must utilize an effective SQL parser to analyze SQL embedded inside Power Query M code.

The SQL parser should extract:

- SQL query name
- SELECT columns
- SQL aliases
- Source tables/views
- Table aliases
- Joins
- WHERE filters
- CASE statements
- Calculated expressions
- Date conversions
- Concatenations
- Arithmetic transformations
- Source column lineage
- Query-level assumptions

The SQL parser should map SQL expressions to the six output columns wherever possible.

For example:

- SQL alias should map to Target Column
- SQL source table/view should map to Source Object
- SQL source field or expression should map to Source Column
- SQL expression should map to Transformation Logic at Column Level
- WHERE clause and join assumptions should be summarized in Notes

The SQL parser should support common patterns such as:

- SELECT a.loan_skey
- SELECT a.loan_skey AS LOAN_SKEY
- CONVERT(varchar, alert_date, 101) AS Alert_Date
- CASE WHEN document_skey IS NULL THEN 'NO' ELSE 'YES' END AS DOCUMENT_ATTACHED
- loan_balance / maximum_claim_amount AS MCA
- CONCAT(LEFT(fha_case_number,3), '-', RIGHT(fha_case_number,7)) AS FHA_Case_Number

If SQL parsing is not 100% conclusive, the application should create a reasonable mapping assumption and clearly state the assumption in Notes.

Mapping Generation Priority

Generate mappings in this priority order:

Priority 1: SQL / Power Query Mapping

Use SQL extracted from Power_Query_M_Code and Data_Sources.

This should be the strongest lineage source.

For each SQL SELECT item:

- Target Object = mapped query output sheet, Excel table, or report tab
- Target Column = SQL alias if available, otherwise selected source column
- Transformation Logic at Column Level = SQL expression or direct mapping
- Source Object = SQL table/view
- Source Column = source column or expression
- Notes = query-based mapping, including filters/joins/assumptions

Priority 2: Header-Based Mapping

Use the Headers sheet to fill gaps where SQL does not explicitly provide final report headers.

For columns with no direct SQL match:

- Match by normalized column name
- Match by business key
- Match by workbook sheet context
- Match by learned sample-output pattern
- Clearly mark uncertain mappings in Notes

Priority 3: Formula-Based Mapping

Use Formulas_Sample for calculated target columns.

For each unique formula:

- Target Object = formula sheet name
- Target Column = formula header name
- Transformation Logic at Column Level = readable or normalized Excel formula
- Source Object = referenced sheet/workbook, if detected
- Source Column = referenced fields, if detected
- Notes = formula-derived mapping

Priority 4: Lookup-Based Mapping

Detect lookup formulas and cross-sheet references such as:

- VLOOKUP
- XLOOKUP
- INDEX/MATCH
- External workbook references
- Translation sheet lookups

Generate business-readable logic such as:

Lookup from <source sheet/workbook> using <key column>

Priority 5: Conservative Inferred Mapping

Use inference only where explicit lineage is unavailable.

Inference should be based on:

- Matching column names
- Matching business keys
- Related workbook tabs
- Query-to-sheet relationships
- Sample output structure learned during development

Mark as:

Inferred from metadata context; review recommended

Deduplication Rules

Avoid over-generation.

Deduplicate using:

Target Object + Target Column + Source Object + Source Column + Normalized Transformation Logic

Additional rules:

- Do not create one row per formula cell.
- Do not create repeated rows for the same copied-down formula.
- Do not create duplicate rows from both SQL and header metadata unless the transformation logic is different.
- Do not generate rows for every metadata field unless it contributes to ETL mapping.
- Keep output concise, traceable, and business-readable.

Verification and Coverage Check Requirements

Add a strong verification layer in the Streamlit application and backend to ensure the generated ETL Mapping Specification is complete and aligned with the uploaded metadata workbook.

1. Metadata Header vs Generated Mapping Pivot Check

The Streamlit application must show a verification pivot/table directly on the frontend after generation.

This verification should compare:

A. Metadata Header Pivot

Read the Headers sheet from the uploaded metadata workbook and create a pivot summary showing:

- Metadata Sheet Name
- Number of Columns Present in Metadata
- Column Names Present in Metadata

This represents the expected report objects and columns available from the metadata.

B. Generated ETL Mapping Pivot

Create another pivot from the generated ETL Mapping Specification output showing:

- Target Object
- Number of Target Columns Generated
- Target Column Names Generated

This represents what the application actually generated in the ETL Mapping Specification document.

The application should display both pivots in Streamlit so the business user can compare metadata coverage against generated mapping coverage.

2. Matching Rule

The metadata sheet names and generated target objects should be compared using normalized names.

For example:

- Trim spaces
- Ignore case
- Replace underscores with spaces for comparison
- Remove special characters where needed

The column names should also be compared using normalized matching logic.

The counts and names should match between:

- Metadata Headers sheet columns
- Generated ETL Mapping Specification Target Column values

3. Skipped Object Rule

Certain sheets/objects can be skipped from the match validation because they are not direct ETL target objects.

The following object names or similar names can be excluded from the matching check:

- Pivot Table
- Pivot Tables
- Dashboard
- Summary
- Chart
- Charts
- Report Results
- Aggregate
- Aggregation
- Lookup only
- Empty sheets
- Metadata-only sheets

These skipped objects should not be treated as missing from the generated ETL Mapping Specification.

However, skipped objects should be captured in the verification output with a note explaining that they were excluded because they are derived, summary, dashboard, pivot, or non-base-data objects.

4. Mismatch Explanation Sheet

If the metadata header pivot and generated ETL mapping pivot do not match, the application must create a separate sheet in the output Excel file named:

Coverage Exceptions

This sheet should explain all mismatches clearly.

The Coverage Exceptions sheet should contain columns such as:

1. Metadata Sheet Name
2. Generated Target Object
3. Metadata Column Name
4. Generated Target Column
5. Exception Type
6. Expected Count
7. Generated Count
8. Reason / Explanation
9. Recommended Action

5. Exception Types

Use the following exception categories:

- Missing Target Object
- Missing Target Column
- Extra Generated Target Object
- Extra Generated Target Column
- Column Count Mismatch
- Skipped Derived Object
- Name Normalization Match
- Needs Review

6. Explanation Logic

For every mismatch, provide a business-readable explanation.

Examples:

- Column is present in metadata Headers sheet but was not generated in ETL Mapping Specification because no SQL, formula, or source lineage was found.
- Target object was skipped because it appears to be a dashboard, pivot table, summary, or aggregate-only sheet.
- Column was generated from SQL alias but exact column name was not found in metadata Headers sheet; review recommended.
- Column count mismatch found between metadata sheet and generated target object. Some columns may be formula-derived, lookup-derived, or excluded due to missing lineage.
- Column name appears matched after normalization, but original names are different.
- Generated mapping row is inferred from metadata context and should be reviewed by SME.

7. Streamlit Frontend Verification Display

After the user clicks Generate ETL Mapping Specification, the Streamlit frontend should show:

1. Preview of generated ETL Mapping Specification
2. Metadata Header Pivot
3. Generated ETL Mapping Pivot
4. Coverage Match Summary

The Coverage Match Summary should show:

- Total metadata target objects considered
- Total metadata columns considered
- Total generated target objects
- Total generated target columns
- Number of matched objects
- Number of matched columns
- Number of skipped derived objects
- Number of missing columns
- Number of extra generated columns
- Overall coverage percentage

Use simple success/warning/error indicators:

- Show success if counts and column names match after allowed exclusions.
- Show warning if there are skipped objects or inferred mappings.
- Show error if there are missing target objects or missing target columns.

8. Excel Output Sheet Rules

The generated Excel output should contain:

1. ETL Mapping Specification
2. Coverage Exceptions, only if mismatches exist
3. Coverage Summary, optional but recommended

The main ETL Mapping Specification sheet must still contain exactly the six required columns:

1. Target Object
2. Target Column
3. Transformation Logic at Column Level
4. Source Object
5. Source Column
6. Notes

Do not add verification columns into the main ETL Mapping Specification sheet.

Any mismatch, count difference, or explanation should go into the separate Coverage Exceptions sheet.

9. Backend Validation Logic

Create backend functions to:

- Build metadata header pivot from the Headers sheet
- Build generated mapping pivot from the final ETL Mapping dataframe
- Normalize sheet names and column names for comparison
- Exclude allowed non-target objects such as dashboard, pivot, summary, and chart sheets
- Compare object counts and column counts
- Identify missing and extra columns
- Generate explanation rows
- Return coverage summary metrics to the Streamlit frontend
- Write Coverage Exceptions sheet into the output Excel if mismatches exist

The validation should be deterministic and should not silently ignore mismatches.
